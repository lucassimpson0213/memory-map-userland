Suggested Learning Order

1
Environment + Observability
Linux tools: perf, htop, iostat, vmstat, strace. Rust tooling: tracing, cargo-nextest, criterion, bacon.
Goal: make sure every project can be run, measured, and debugged from day one.
Outputs: working perf setup, tracing logs, benchmark harness, checklist of core Linux tools.
2
Core Systems Data Structures
Ring Buffer, Bitmap Allocator, Hash Table, Priority Queue, B-Tree.
Goal: learn memory layout, cache behavior, and throughput tradeoffs.
Outputs: minimal implementations, service wrappers where useful, measurement notes.
3
First System
In-memory KV store with service runtime and observability.
Goal: combine data structures into a real service.
Invariants: reads after writes return latest committed value; requests are measurable.
4
Durability Layer
Write-Ahead Log (WAL).
Goal: persist mutations before applying them.
Invariants: once fsync completes, committed entries are never lost.
5
Crash Recovery
WAL replay and state reconstruction after restart.
Goal: recover consistent state after failure.
Outputs: replay procedure, recovery tests, crash injection notes.
6
Networking
TCP server, request protocol, CS144.
Goal: move from local service to networked system.
Outputs: request protocol, client/server flow, network-level metrics.
7
Concurrency
Multi-threaded or async requests, locking and contention analysis.
Goal: handle multiple requests safely and efficiently.
Focus: contention, queue buildup, tail latency, race conditions.
8
Replication
Multi-node KV store, reinforced by Fly.io Maelstrom.
Goal: preserve correctness across nodes.
Invariants: duplicated or delayed messages do not corrupt state; nodes converge correctly.
9
Cloud Architecture
Optional: LocalStack, S3 snapshots, SQS queues, workers.
Goal: map system ideas into cloud architecture patterns.
Outputs: snapshot pipeline, queue-driven workers, orchestration experiments.

Rule: Only one milestone active at a time.

Performance Debugging Framework

RED (Service Level)
  • Rate → requests/sec
  • Errors → error rate
  • Duration → latency (p50 / p95 / p99)

Ask: Is throughput dropping? Are errors increasing? Is latency rising?

USE (System Level)
  • Utilization → how busy is it?
  • Saturation → are requests waiting?
  • Errors → are failures occurring?

Inspect CPU, Memory, Disk, Network, and Locks / threads.

Profiling (Code Level)
  • profile CPU
  • inspect hot paths
  • measure allocations

Tools: perf, flamegraphs, cargo bench, criterion.

Observability Layer

  • structured logs
  • latency metrics
  • throughput metrics
  • error counters
  • backlog / queue depth metrics
  • traces for multi-step operations

Use both application-level observability and system-level Linux tools.

Rule: A component is not complete until you can observe, measure, and explain its behavior under normal load, stress, and failure.

Service Runtime + Long-Running Tests

  • Expose a service interface when reasonable
  • Run for extended periods
  • Observe memory, CPU, disk, and latency over time
  • Restart it and verify recovery behavior
  • Inject failures while running
  • Confirm invariants still hold after long execution

Test goals

  • detect memory leaks
  • detect queue buildup
  • detect latency drift
  • detect retry storms
  • detect crash recovery issues
  • verify durability under extended runtime

Failure Laboratory

Your Raspberry Pi server is a reliability lab. Install Cockpit on both servers.

  • kill process during writes
  • crash during job execution
  • restart services mid-operation
  • restore from backups

Goal: develop intuition for real durability guarantees.
